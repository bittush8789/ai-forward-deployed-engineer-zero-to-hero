# Module 7.6: Slowly Changing Dimensions (SCD)

Welcome to **Slowly Changing Dimensions (SCD)**. In real-world enterprise databases, dimension data changes over time. A customer might change their email, move to a new state, or upgrade their subscription tier. In this module, you will learn the four core types of Slowly Changing Dimensions used to manage these updates while preserving historical query accuracy.

---

## 1. Detailed Theory

### SCD Types
SCD defines the rules for handling modifications in dimension tables:
1. **SCD Type 1 (Overwrite)**:
   - The old value is simply overwritten with the new value. No historical data is preserved.
   - Use Case: Correcting typos (e.g., fixing a misspelled name).
2. **SCD Type 2 (Historical Tracking)**:
   - The gold standard for data warehousing. A new row is inserted to record the change.
   - Requires tracking columns: `is_current` (boolean), `valid_from` (timestamp), and `valid_to` (timestamp).
   - Use Case: Tracking customer relocations to ensure historical sales are mapped to the correct state at the time of purchase.
3. **SCD Type 3 (Previous Value Tracking)**:
   - Add a column to store the previous value. Only tracks the current and immediate previous state.
   - Use Case: Tracking immediate status changes (e.g., `current_status` and `previous_status`).
4. **SCD Type 4 (Historical Tables)**:
   - The active record is kept in the main dimension table (which behaves like Type 1), while all historical updates are appended to a separate history table.

## 3. Production Use Cases

1. **Customer History Tracking System**: A company wants to analyze user subscription tier upgrades over 3 years. They configure `dim_customers` as an SCD Type 2 table. When a user upgrades from "Basic" to "Premium," a new row is created. This allows analysts to calculate how much revenue the user generated while in the "Basic" tier vs. the "Premium" tier.

## 5. Coding Examples

### Implementing SCD Type 2 in SQL (MERGE Statement)

This SQL query shows how to execute an SCD Type 2 update on a target dimension table using staging updates.

```sql
-- Step 1: Terminate the active flag for existing customers whose email changed
UPDATE dim_customer
SET 
    is_current = FALSE,
    valid_to = CURRENT_TIMESTAMP
WHERE customer_id IN (SELECT customer_id FROM staging_customer)
  AND is_current = TRUE;

-- Step 2: Insert the new active records for the updated customers
INSERT INTO dim_customer (customer_id, email, is_current, valid_from, valid_to)
SELECT 
    customer_id,
    email,
    TRUE AS is_current,
    CURRENT_TIMESTAMP AS valid_from,
    '9999-12-31 23:59:59'::TIMESTAMP AS valid_to
FROM staging_customer;
```

## 8. Interview Questions

1. **What is an SCD Type 2 and how do you configure it?**
   *Answer Hint: Slowly Changing Dimension Type 2 tracks historical changes by inserting a new record for every modification of a natural key. It requires generating new surrogate keys and using tracking columns: an active flag (is_current) and validity date ranges (valid_from/valid_to).*
2. **Why is SCD Type 1 dangerous for analytical reporting?**
   *Answer Hint: Type 1 overwrites old values. If a customer moves from New York to California, and you overwrite the state column, all their historical purchases in New York will be counted as California sales, distorting historical regional reports.*

---

## 9. Best Practices (FDE Standards)

- **Always use validity date ranges**: When implementing SCD Type 2, always include both `valid_from` and `valid_to` timestamps, using a far-future default date (like `9999-12-31`) for active rows to facilitate SQL `BETWEEN` joins.
- **Automate with dbt**: Do not write manual SQL update/insert scripts. Use dbt's built-in snapshots feature (`dbt snapshot`) to automate SCD Type 2 tracking.

---

## 10. Common Mistakes

- **Using Natural Keys as PKs**: Attempting to run SCD Type 2 on a table where the natural key (`email`) is the primary key, causing unique constraint violations when a new history row is inserted.
- **Overlapping Dates**: Having gaps or overlaps in the `valid_from` and `valid_to` timestamps during updates, resulting in duplicate joins in fact tables.
